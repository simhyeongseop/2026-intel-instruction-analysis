## 🔨 어셈블러 테스트 코드 생성 (Random Generation)

### 개요
그룹화가 완료되면 컴파일러(어셈블러)가 매뉴얼을 제대로 지키는지 확인하기 위한 테스트 코드를 만듭니다.

### 랜덤 예제 생성
- 추출한 그룹(예: R64) 자리에 **실제 레지스터**(예: RAX, RBX)나 값을 **랜덤하게 넣어** 어셈블리 코드를 라인별로 생성합니다.

### 인텔 문법 명시 ⭐
- 컴파일러가 **AT&T 문법이 아닌 인텔(Intel) 문법**으로 인식하도록 테스트 코드 **첫 줄에 반드시** 다음을 추가해야 합니다:
  ```asm
  .intel_syntax noprefix
  ```

### 생성 횟수
- 모든 경우의 수를 테스트하기는 어려우므로 **약 10번 정도 랜덤하게 생성**하여 테스트를 진행하는 것을 권장합니다.

## 🔧 컴파일 및 오류 분석 (에러 Triage)

### 컴파일 실행
만들어진 코드를 실제로 컴파일하고 발생하는 에러를 수집합니다.

**컴파일 명령어**:
```bash
gcc -c <src> -o <output>
```
오브젝트 파일 컴파일을 시도합니다.

### 에러 파싱 및 주석
- 컴파일 시 발생하는 **엄청난 양의 에러 메시지**를 확인합니다.
- **에러가 발생한 라인 번호**와 **에러 내용**을 추적하여 코드에 **주석으로 남기며 수정**합니다.

### 오류 그룹화 (Triage) ⭐
- `unsupported instruction`, `unknown instruction` 등 **비슷한 에러들끼리 묶어서 정리**합니다.
- **다음 미팅 때** 자주 발생하는 에러의 **대표 예제 2~3개**를 가져와서 논의합니다.

### 크로스 체크 (컴파일러 비교)
- **GCC**뿐만 아니라 **Clang 컴파일러**로도 테스트해 봅니다.
- 두 컴파일러의 결과가 다르게 나오는지를 비교합니다 (예: 레지스터가 몰래 바뀌어 컴파일되는지).
- **컴파일러의 버그를 찾을 수 있습니다**.

## 📋 오퍼랜드 종류별 예제 생성 규칙 (중요)

### 상수값 (Immediate, IMM)
- **IMM32**나 **IMM16** 같은 상수값은 경우의 수가 너무 많으므로:
  - ❌ 무작위로 뽑지 말 것
  - ✅ **해당 크기의 최대값**(예: `0xFFFF`, `0xFF`)을 **16진수로 넣어** 테스트합니다.

### 메모리 접근식 (Intel Memory Addressing)
레지스터나 메모리로 접근할 때는 아래 두 가지 방식을 **섞어서 케이스를 만듭니다**:

#### 1. Effective Addressing
```asm
[base + index * {1,2,4,8} + disp]
```
예제: `[rax + rbx*4 + 0x100]`

#### 2. RIP-relative Addressing
```asm
[rip + disp]
```
예제: `[rip + 0x1000]`

### 메모리 크기 명시 (PTR) ⭐
메모리에 접근할 때는 **크기를 명시하기 위해** 접근식 앞에 **지시어를 반드시 붙여야** 합니다:

| 크기 | 지시어 | 사용 예 |
|------|--------|--------|
| 8비트 | `BYTE PTR` | `mov al, BYTE PTR [rax]` |
| 16비트 | `WORD PTR` | `mov ax, WORD PTR [rax]` |
| 32비트 | `DWORD PTR` | `mov eax, DWORD PTR [rax]` |
| 64비트 | `QWORD PTR` | `mov rax, QWORD PTR [rax]` |
